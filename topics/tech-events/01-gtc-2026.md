## 前言：從實驗室到 AI Factory

「教授，」小陳一邊關掉投影，一邊看著白板上的「GTC 2026」幾個大字，「這一屆聽起來好像什麼都變成 AI Factory 了，我應該先看什麼？」

我笑了一下，拿起白板筆畫了三個圈：**Vera Rubin 平台**、**Disaggregated Inference**、**AI Factory / Physical AI**。

「如果你把這三個圈看懂，」我說，「你就不會只看到『又多了一顆更快的 GPU』，而是看到整個產業從實驗室走向工廠的轉折。」

從工程師的角度看，GTC 2026 很清楚地傳遞了一個訊號：AI 產業已經從「實驗階段」走進「工業部署」。NVIDIA 不再只是賣單一 GPU 晶片，而是把自己定位成一個完整的 **AI 基礎設施平台**——從 CPU、GPU、網路、儲存到整個機櫃與資料中心設計，統一包裝成可以直接部署的「AI 工廠」。

這一代的主角不是單一晶片，而是一整個系統。Vera Rubin 平台把 Vera CPU、Rubin GPU、NVLink 6、HBM4、光互連與 rack‑scale 設計綁在一起，對外交付的是「一櫃就能開工的 AI Factory」。

在這個脈絡下，Vera CPU 的角色特別關鍵。過去十多年，我們習慣把 CPU 看成「通用管理者」：負責作業系統、排程與 I/O 協調，真正吃重的 AI 計算交給 GPU。Vera CPU 則往前跨了一步——它不只是「支援 AI」，而是被設計成 **直接驅動 Agentic AI 的推理中樞**。

根據 GTC 2026 的 session 資料與後續報導，NVIDIA 將 Vera CPU 描述為一顆專門為 agentic loops 打造的 custom Arm 核心：用確定性延遲（deterministic execution）與極低的 orchestrator latency，解決過去「CPU 變成整個推理 pipeline 瓶頸」的問題。有媒體以「一次可以驅動 22,500 個 AI agent」來形容這種規模與併發能力——實際數字仍需以官方文件為準，但方向很清楚：**這顆 CPU 是為大規模、多代理的 AI 系統量身打造**。

接下來，我們會先從 Vera Rubin 平台開始，說明為什麼 NVIDIA 需要一顆新的 CPU，才能真正把 GPU、網路與記憶體組合成一座可長期營運的 AI Factory。

## 1. Vera Rubin 平台：為 Agentic AI 設計的下一代異構架構

「教授，」小楊把筆電轉向我們，「這張 NVIDIA roadmap，我總覺得從 Blackwell 到 Vera Rubin 的跳躍有點奇怪。如果只是把 GPU 做更快，為什麼還要多設計一顆 Vera CPU？」

「你把時間軸往前拉一點看，」我在白板上畫出幾個世代的關鍵節點，「這不是單純多一顆 chip，而是重新分工：誰負責思考、誰負責計算。」

下面這一節，就從 roadmap 開始，把這個分工講清楚。

### 1.1 Roadmap 與大局：從 Blackwell 到 Vera Rubin

從架構演進的角度看，Vera Rubin 不是一顆「突如其來的新晶片」，而是整個 roadmap 上的下一個階段：

- **Blackwell (2024–2025)**：
  - 以 B200 / GB200 為核心的 GPU 平台
  - 典型機櫃功率已經逼近 120kW+，在許多資料中心裡接近電力與冷卻的上限
- **Vera Rubin (Late 2026)**：
  - 在算力上延續 Blackwell 之後的世代
  - 更重要的是：重新設計 CPU、GPU、記憶體與互連的分工
  - 把「推理控制與資料搬移」提升為一級設計目標，而不是 GPU 邊上的附屬功能
- **Feynman (2027+)**：
  - 更遠期的 roadmap 節點
  - 在本篇文章中只作為長期方向的背景，不做細部推測

把這條時間線整理成一個簡單的表格，可以更直觀地看到焦點如何轉移：

| 時間        | 平台       | 說明                           |
|------------|------------|--------------------------------|
| 2024–2025  | Blackwell  | 把 GPU 推到現有機房極限       |
| 2026 (Late)| Vera Rubin | 把 GPU + CPU + 互連變成整體平台 |
| 2027+      | Feynman    | 更遠期、仍在規劃中的架構       |

小楊看著這張圖，皺著眉頭問：「教授，這看起來還是很像正常的『新製程、新 GPU』節奏啊。為什麼你說這是架構層級的改變，而不是單純多一顆晶片？」

「關鍵不在名字，」我指了指白板，「而在**背後的應用和工作負載指紋變了**。這就是我們在《Computer Architecture 02/03》和 *System Design — An Architecture-Aware Approach* 一再強調的：**架構不是從晶片開始設計，而是從 workload 往回推。**」

如果用那套方法論來看，這三個節點可以粗略對應到三個不同的應用時代：

| 世代       | 主力應用 / 業務模式            | 工作負載指紋                              | 架構設計焦點                 |
|------------|--------------------------------|--------------------------------------------|------------------------------|
| Blackwell  | 超大模型訓練 + 推理起飛        | 長時間 batch 訓練 + 初代線上推理           | 「讓單櫃 GPU 更快」          |
| Vera Rubin | 大規模線上推理 + Agentic AI   | 高併發、長 context、agentic loops          | 「把推理控制拉成一級公民」  |
| Feynman    | Physical AI / Digital Twin    | 實體迴圈、safety‑critical                  | 「從資料中心延伸到實體世界」 |

這個 roadmap 透露出一個關鍵訊息：

> **NVIDIA 不再只是在「疊更大的 GPU」，而是在重新定義整個推理系統裡誰負責「思考」、誰負責「計算」。**

用 *System Design* 那本書裡的語言來說，Blackwell 世代之前，整個系統的「設計旋鈕」還主要集中在 GPU：

- 透過更多 SM、更多 Tensor Core、更多 HBM channel，把 Roofline 的 compute/帶寬上限往上推；
- CPU 更像是《03 篇：Workload‑driven CPU selection》裡說的「host/管理核心」——重要，但多半不在 hot path 上。

但從 Blackwell 世代開始，我們就已經看到 inference 的壓力快速上升：大型語言模型被投入產品線，使用者數量級成長，延遲與可用性變成一級指標。到了 Vera Rubin，這個壓力被重新整理成幾個非常具體、而且是**應用驅動**的設計問題：

- **成本曲線**：在不爆掉功耗與 TCO 的前提下，如何支撐 7x24 的長期推理負載？（Queuing + Little's Law 視角）
- **agentic 行為**：當一個應用需要成百上千個 agent 長時間協作時，誰負責維持這些 agent 的狀態與行為？（Amdahl + USL 視角）
- **long‑context / 多階段 pipeline**：當推理從「單次問答」變成「長時間任務編排」時，系統的瓶頸還會只是在 GPU 上嗎？還是會轉移到記憶體、網路與 orchestrator 本身？（Roofline 視角）

換句話說，從 *processor‑centric* 走向 *AI‑factory‑centric*：

- 在 Blackwell 時代，你可以勉強把問題簡化成「要幾顆 GPU、每顆多快」；
- 在 Vera Rubin 時代，你必須同時設計「**誰 orchestrate、誰 compute、誰搬資料，以及這三者的延遲與 jitter 分佈**」。

在這個新的 regime 裡，**orchestrator 的效率與資料搬移成本，往往比單純多一點 FLOPs 更早把系統推到瓶頸**，也替下一節 Vera CPU 的登場埋好了動機。

Vera Rubin 平台的回答之一，就是在 GPU 旁邊放上一顆專門為這種工作設計的 CPU——Vera CPU，
把我們在《02：異構系統架構》裡講過的那句話真正落實到矽上：

> **讓對的核心處理對的工作負載，而不是強迫一種核心撐起所有角色。**

### 1.2 Vera CPU：為 Agentic AI 打造的確定性執行核心

#### 從「通用管理者」到「推理中樞」

在傳統的異構系統裡，CPU 的角色很清楚：

- 負責作業系統與 runtime
- 負責排程、I/O 與網路協調
- 把大量矩陣運算 offload 給 GPU

這種分工在「離線訓練」階段運作得不錯，因為訓練 workload 的 latency 容忍度相對高，重點是 throughput 與成本。但在 **Agentic AI** 與大規模推理的世界裡，這樣的 CPU 角色開始不敷使用：

- 多個 agent 同時運行、彼此呼叫工具與服務
- 每個 agent 需要維持自己的 context 與長期狀態
- 系統必須在毫秒到秒級的時間內做出決策、執行動作、更新世界狀態

在這樣的情境下，CPU 不再只是「旁邊幫忙的管理員」，而是整個系統的 **reasoning engine / orchestration engine**：

- 決定哪個 agent 先執行、哪個延後
- 決定哪些任務要送到 GPU、哪些在 CPU 就地處理
- 決定什麼時候該等待 I/O、什麼時候可以樂觀前進

如果這一層的延遲是不可預測的，整個系統的行為就會變得很難推理、很難除錯，甚至很難給出任何可靠的 SLO（Service Level Objective）。這就是為什麼 Vera CPU 會被設計成一顆 **強調確定性執行（deterministic execution）** 的核心。

#### 技術設計：為 agentic loops 最佳化

在《Computer Architecture 03：Workload‑driven CPU selection》裡，我們把 Arm 的 CPU 大致分成兩個世界：

- **Cortex 系列**：手機、筆電、消費端 SoC（X 系列大核、A 系列中/小核），追求的是「使用者體感上的瞬間爆發力」。
- **Neoverse 系列**：資料中心與 HPC（N/V 系列），追求的是「全年無休的吞吐量與可擴展性」。

這兩個世界大多共享相同的 **Armv9‑A ISA**——也就是說，程式看到的指令集幾乎一樣。但在微架構層級，Arm 會根據不同 workload，調整幾顆關鍵旋鈕：

- **Decode 寬度**：一次能「聽懂」幾條指令
- **重排序緩衝區（ROB）與 in‑flight 容量**：能同時在飛行中的指令有多少（Little's Law：`In‑Flight = IPC × Latency`）
- **向量引擎（SVE2）寬度**：一次能處理多少資料元素
- **Cache / L2 / L3 容量與延遲**：能否撐住大型、pointer‑chasing 的工作集

這個簡化版 Little's Law 給了一個很實用的直覺：

> **In‑Flight ≈ IPC × Latency**
> *（系統中同時在飛行的指令數 ≈ 每週期指令吞吐量 × 單一指令的平均延遲）*

從微架構的角度看，**大型 Out‑of‑Order 引擎與 aggressive branch predictor 本身就是一把雙面刃**：一方面可以拉高峰值 IPC，另一方面也會把每一次 L3 miss、TLB miss 或 branch mispredict 的「懲罰」，放大成整條尾端的 jitter。

當你把 decode 做得很寬、ROB 開得很大，等於是讓更多指令同時「在空中飛行」，期待換到更高的 IPC。只要 pipeline 裡有幾個延遲波動很大的元件（例如 L3 miss、TLB miss、遠端記憶體或 I/O），這些 in‑flight 指令就會在那幾個點前面排隊，整條尾端的完成時間被一起往後推，形成長尾延遲。

對手機 SoC 來說，偶爾出現幾次長尾延遲不一定致命；但對 24×7 的 agentic 系統與 Physical AI 場景，我們關心的是 **P95 / P99 延遲是否穩定**，而不是單一核心的峰值 IPC。這也是為什麼 Vera CPU 在設計上，更接近 real‑time / safety‑critical 世界的哲學：

- 把 speculation 深度與最複雜的 branch predictor 路徑壓在可控範圍內；
- 用比較「平坦」的 cache 階層換取更可預期的延遲分佈；

讓整個系統在面對排隊與抖動（jitter）時，比較不會突然掉進 Queuing Theory 裡的「排隊黑洞」。

如果用 2026 年的 Arm 產品來舉例，可以粗略畫出這樣一張對照表（結合 Arm 官方白皮書與 Hot Chips 報告的公開資訊）：

微架構對照（概念級，非逐位規格）

| 特性       | Cortex‑X925 (手機大核)       | Cortex‑A725 (高效大核)       | Neoverse V3 (伺服器/HPC)                 |
|------------|-----------------------------|------------------------------|------------------------------------------|
| 設計目標   | 極致單核 Burst IPC          | 能效與多核平衡               | 資料中心吞吐量 / 可擴展性                |
| Decode 寬度| ~10‑wide                    | ~4–5‑wide                    | ~6–8‑wide                                |
| ROB 規模   | ~768+ entries               | 中等（數百）                 | 大（約 300–400）                         |
| L2 / L3    | 中等、偏向單核延遲          | 較小、節省面積               | 大容量、低延遲、共享                     |
| 向量引擎   | SVE2，偏 128‑bit 應用       | SVE2，能效優先               | SVE2，256/512‑bit、對標 HPC/AI          |
| 擴展方式   | 1–2 顆大核 + 多顆小核       | 4–8 核 SoC                   | 64–128+ 核、支援 CXL/PCIe 6             |

你可以把這張表想成是《03 篇》裡「選布料」那張圖在 Arm 世界的具體化：

- X925 像是旗艦手機裡的「超跑引擎」，在極短時間內拉到最高 IPC；
- A725 是長時間跑馬拉松的「高效能大核」，追求的是 Perf/Watt；
- Neoverse V3 則是資料中心裡的「重工業馬達」，重點是 7x24 小時穩定吞吐與 TB/s 級記憶體系統。

根據 GTC 2026 session [CWES81771] 的說明，Vera CPU 採用的是 NVIDIA 自家設計、以 **Neoverse V 系列級別核心** 為基礎的 Arm 客製微架構，而不是把手機裡的 X 系列大核搬進機房。這裡的設計思路，很明顯是對著「agentic 推理迴圈」這種 workload 來調整：

- 不追求像 Cortex‑X925 那樣極端寬的 10‑wide decode，而是在 **6–8‑wide + 大 ROB** 之間取得平衡：既能挖出足夠的 ILP/MLP，又不讓每顆核心的功耗失控；
- 把更多晶體管預算放在 **L2/L3 cache、預取器與記憶體階層** 上，對應 CA02 裡談過的 Memory Wall 與 Roofline 限制；
- 強化 **SVE2 向量引擎與 CXL 3.0 / PCIe Gen 6 介面**，讓它同時能扮演 GPU/DPU/NPU 的高效 host。

換個角度說：

> **Vera CPU 不是「更大的手機大核」，而是一顆「為資料中心 agentic workload 重新調過旋鈕的 Neoverse 級客製核心」。**

在這個前提下，[CWES81771] 提到的幾個具體設計方向就比較好理解了：

- 簡化最容易製造尾延遲（tail latency）的 branch predictor 路徑與過度複雜的 cache 階層，換取更好的延遲分佈穩定性；
- 在前端與排程器加入對 **agentic loops（規劃 → 查詢 → 執行 → 反饋 → 再規劃）** 的優化，把這類 control‑heavy 工作視為一級 workload，而不是一般「背景服務」；
- 在叢集層級，透過更細緻的 QoS / rate‑limiting 機制，避免 orchestrator 自己成為 Queuing Theory 裡的「排隊黑洞」。

這種設計思路與近年興起的 LPU（Language Processing Unit）有明顯呼應，但落點不同：

- LPU 強調靠 on‑chip SRAM 提供極低延遲，適合 **小模型 + 超高併發** 的固定拓樸推理；
- Vera CPU 則是站在一個更「系統級」的位置：
  - 利用 HBM4 與 Rubin GPU 的高頻寬記憶體作為後盾，承接大型模型與長 context；
  - 在 CPU 端提供穩定、可預期的控制流與資料搬移，成為 agentic 系統的「中樞神經」。

媒體用「一次可以驅動 22,500 個 AI agent」來形容 Vera CPU 的規模與併發能力，某種程度上反映了這種設計目標：

> **它不是單純把舊有 CPU 做得更快，而是把 CPU 當成一顆專門為大量 AI agent 提供推理與協調服務的處理器。**

實際上，Vera CPU 真正重要的並不是單一 core 的峰值性能，而是整個 platform 在以下幾點上的表現：

- 在大量 agent 同時進行工具呼叫與網路 I/O 時，延遲是否仍然穩定？（Queuing Theory + tail latency）
- 在 GPU 滿載的情況下，orchestration 邏輯是否會被「卡在 CPU」？（Amdahl：CPU 串行部分不能變成瓶頸）
- 是否能提供足夠可預期的行為，讓系統架構師可以對 SLO 和 cost 進行設計與驗證？（USL：控制擴展時的 κ 係數）

這些問題，正是 Vera CPU 相對於上一代平台（如 Grace + Blackwell）試圖解決的核心。

如果用更「架構比較」的方式來看，Vera CPU 也可以被視為對近年 LPU 類架構的一種回應：

- **LPU 類架構**：用大量 on‑chip SRAM 換取極低延遲，適合小模型、高併發、拓樸相對固定的推理系統。
- **Vera CPU + Rubin GPU**：用 HBM4 與 rack‑scale 互連，在略高但可預期的延遲下，換取更大的參數容量與更彈性的部署拓樸。

我們可以用一個簡化的表格來對照兩種設計的 trade‑off（概念示意）：

| 設計              | 延遲       | 記憶體容量           | 拓樸彈性                 |
|-------------------|------------|----------------------|--------------------------|
| LPU (SRAM 為主)   | 極低       | 受限於單顆晶片       | 以 scale‑up 為主         |
| Vera + Rubin      | 低且可預期 | HBM4 + rack‑scale    | 支援大規模 scale‑out     |

在 GTC 2026 的工程分析材料裡，這條路線被總結成一個目標區間：

> **在追求非常大的模型與長 context 的同時，把每 token 延遲壓在大約 sub‑100ms 的等級，並維持足夠的確定性，讓整個 agentic 系統好設計、好除錯。**

### 1.3 Rubin GPU：HBM4 與 rack‑scale 設計

如果說 Vera CPU 解決的是「誰來思考、誰來協調」，那 Rubin GPU 面對的則是另一個現實問題：**當模型與工作負載愈來愈大時，單顆 GPU 本身的效能早已不是唯一瓶頸，真正的限制來自記憶體與互連**。

在 Vera Rubin 平台中，Rubin GPU 的幾個關鍵變化可以簡化成三點：

- **HBM4 記憶體**：在頻寬與容量上大幅提升，縮小與 SRAM‑based 晶片（如部分 LPU 架構）之間的差距。
- **rack‑scale 互連設計**：NVLink 6 提供更高的跨 GPU 頻寬，讓多顆 Rubin 在同一個 NVLink domain 裡能更有效地共享 KV‑cache 與中間特徵。
- **與光互連的結合**：在跨櫃擴展時，透過 CPO（Co‑Packaged Optics）與新一代 switch，降低每 bit 功耗，讓 72 → 144 → 576 顆 GPU 的擴展仍然可行。

如果把這些變化換成頻寬上的數字（概念示意，實際數值仍以白皮書為準），可以更容易看出這一代的「尺度」：

Rubin vs Blackwell（頻寬視角，約略數字）

| 項目                  | Blackwell (HBM3e / NVLink 5) | Rubin (HBM4 / NVLink 6) |
|-----------------------|------------------------------|-------------------------|
| HBM 記憶體頻寬        | ~ 8 TB/s                     | ~ 22 TB/s               |
| NVLink GPU↔GPU 頻寬   | ~ 1.8 TB/s                   | ~ 3.6 TB/s              |

這裡重點不是小數點後幾位，而是**等級差異**本身：當單顆 GPU 內部的 HBM 與互連頻寬成倍提升時，系統的瓶頸自然從「單卡算力」慢慢移向 **跨 GPU、跨機櫃的資料搬移與拓樸設計**。

> **Arithmetic Intensity（算術強度） ≈ `FLOPs / Bytes`**——每搬 1 Byte 資料，能做多少實際有用的運算。

如果用《Computer Architecture 04》裡的 Roofline 視角來看，可以把這些頻寬想成是把「每秒可搬運的 Bytes」往上拉高，讓 Rubin 在 arithmetic intensity 較高的工作（粗略來說，就是每搬 1 Byte 可以做很多 FLOPs 的那一類）時，有機會更貼近理論上的 compute 上限；
反過來說，當 workload 的 arithmetic intensity 偏低、需要頻繁存取 KV‑cache 或在 GPU 之間來回搬資料時，真正的瓶頸就會從單卡 TFLOPS 轉移到 HBM / NVLink / 機櫃之間的光互連。

第二節我們會用 Prefill / Decode 的對照表，把這件事畫成更具體的 Roofline 圖，說明為什麼「同一顆 Rubin」在不同階段會呈現出完全不同的瓶頸樣貌。

這也是為什麼在談 Rubin 的同時，GTC 2026 會花大量篇幅介紹 Spectrum‑X、Quantum‑X 與 CPO——因為沒有這一層，前面那幾十 TB/s 的本地頻寬，很快就會被機櫃之間的連線吃掉優勢。

在 GTC 2026 的展示中，NVIDIA 已經拿出 Rubin 的實體矽樣品，並給出大致的生產時程（Late 2026）。對工程師而言，更重要的不是單一顆 GPU 的 TFLOPS，而是理解：

- **HBM4 提供的是「prefill 階段」所需的巨大記憶體頻寬與容量**，用來承載 KV‑cache 與中介特徵。
- **NVLink 6 與 rack‑scale 拓樸決定了多顆 Rubin 如何被「視為一個邏輯整體」**，而不是彼此孤立的加速卡。
- **與 CPO / Spectrum‑X / Quantum‑X 結合後，整個系統的瓶頸開始從單櫃內走線，轉移到跨櫃拓樸與機房等級設計**。

這些設計在後面談 Disaggregated Inference 與 AI Factory 時會再出現一次——因為它們不只影響單一應用的效能，而是決定「整個資料中心可以承受多少 inference 負載」。

### 1.4 數值精度與模型層創新

Vera Rubin 平台的另一個關鍵，是在 **數值精度與模型層面** 做出的選擇。這些創新看起來像是「演算法 / 數學細節」，但實際上直接決定了硬體可以在多大程度上「換取」模型大小與推理成本：

- **NVFP4（4‑bit floating point）**：在保持一定動態範圍的前提下，把資料寬度壓到 4 bit，讓同樣的記憶體頻寬可以承載兩倍以上的運算吞吐。
- **1.58‑bit 量化（BitNet）**：把權重壓到三值（{-1, 0, 1}），大幅降低參數儲存與存取成本，讓 edge / embedded 等場景也能跑較大的模型。
- **Sparse Attention Kernels**：針對長 context（例如 128k tokens 以上）的情境，透過稀疏 attention 把 VRAM 壓力降下來，避免 context 長度成為推理成本的致命瓶頸。

這些功能並不是「免費贈品」，而是對應到非常具體的工程問題：

- 在資料中心裡，**推理成本往往由「每 token 需要搬多少資料」決定，而不是單純的算力**。
- 在 edge 或機器人等場景裡，**功耗預算與記憶體容量遠低於機房等級**，必須用更激進的量化方式才能塞進可接受的 envelope。

如果用《Computer Architecture 04》裡的 Roofline 語言來看，當你把權重從 16‑bit 壓到 8‑bit、4‑bit，再壓到 1.58‑bit，實際上同時在做兩件事：

- 對資料中心來說，在相同的 HBM / NVLink 頻寬之上，可以掛上去的 FLOPs 變多，整體 arithmetic intensity 被暴力地往右推，讓原本 memory‑bound 的部分有機會更貼近 compute 上限——本質上就是靠壓縮資料體積，來強行翻越「記憶體牆 (Memory Wall)」，把工作負載推向「算力高原 (compute‑bound 區域)」；
- 對 edge / 機器人系統來說，等於是用更少的 bit 去表達同樣多的資訊，把記憶體 / 頻寬壓力「壓扁」，換成一點可以控制的算術複雜度與模型設計工作。

對讀者來說，這一節的重點不在於背下每一種數值格式，而是理解：

> **當你在選擇模型與部署平台時，其實是在同時選擇「數值精度」與「硬體路徑」，兩者是綁在一起的。**

這也預告了後面主體二（Disaggregated Inference）的一個主軸：不同階段的 inference，適合採用不同的數值格式與硬體組合。

### 1.5 基礎設施需求：800V、液冷與 rack‑scale 系統思維

最後，主體一要補上的，是一個常常在「產品發表」裡被淡化、但在工程現場非常真實的問題：**要讓這樣一整櫃的 Vera Rubin 平台運轉起來，需要什麼樣的電力與冷卻基礎設施？**

在 GTC 2026 的材料裡，可以看到幾個清楚的方向：

- **800V DC 電力方案**：
  - 從傳統的 48V 轉向 800V DC，目的在於降低 I²R 損耗，讓同樣的電纜與配電體系可以承載更高的功率。
  - 這不只是「換一顆電源供應器」，而是整個機房級別的配電設計要重新來過。

- **direct‑to‑chip 液冷**：
  - 隨著單櫃功率與功率密度上升，風冷系統愈來愈難以在合理噪音與空間條件下穩定運轉。
  - Vera Rubin 所對應的資料中心設計，多數都假設使用封閉式液冷迴路與 CDUs（Coolant Distribution Units）。
	- 這樣的液冷與配水設計，不只是為了壓制 Rubin GPU 本身的數十 kW 熱密度，也是為了在機櫃內維持穩定、可控的溫度場，讓旁邊**極度熱敏感的 CPO 矽光引擎（例如常見的 MRM 微環調變器）**不會因為微小的溫度漂移，就頻繁引發致命的網路 jitter 與重傳風暴。

- **20MW 級 AI site 作為設計單位**：
  - 在多場 GTC session 裡，NVIDIA 會以「十幾到幾十個這樣的機櫃」聚合成一個約 20MW 的 AI site，作為實際規劃與投資評估的基本單位。
  - 第三章我們會用一個 Fermi Estimate 的小教室，示範如何從 20MW 的電力預算推回機櫃數量與拓樸選擇。

 技術洞察其實很直接：

> **Vera Rubin 這一代的「架構升級」，已經不可能只在晶片上完成，而是被迫把整個 rack‑scale system（電力、冷卻、互連）當作一個整體來設計。**

 這句話背後隱含的是：你不再只是為「單一機櫃」選電源與風扇，而是在為一整個 10+ MW / 20MW 級別的 AI site 設計配電與冷卻 envelope。

 這也是為什麼 NVIDIA 在談 Vera Rubin 時，會同時談到 NVL72 / Kyber / NVL576 這些「機櫃與叢集等級」的設計——因為對客戶來說，真正買到手的不是一顆 GPU，而是一整個可以開工的「AI site」。

## 2. Disaggregated Inference：從訓練到生產的架構轉變

「教授，」小陳指著螢幕上的 cost 圖表，「我們明明只訓練一次，為什麼帳單每個月都在往上衝？」

小楊在旁邊補充：「我把 request log 抓下來算過，實際上 9 成以上的成本都在長期在線上的 inference workload。」

我請他們把圖放大，只保留 inference 那一條。「這條才是 1 兆美元的主角，」我說，「GTC 2026 在談的，就是怎麼把這條曲線壓下來。」

如果說 Vera Rubin 平台回答的是「**硬體長什麼樣子**」，那 Disaggregated Inference 回答的則是「**為什麼要這樣長，才能撐住未來的推理負載**」。

在《Computer Architecture 01：All roads lead to IPC》與《04：LLM 與 RVV 方法論》裡，我們把這類問題拆成兩個向量：

- **算得多不多（throughput / concurrency）**：一年內平台能處理多少 token？
- **算得值不值得（cost / SLO）**：每 token 成本是多少？在目標延遲內能不能穩定交付？

GTC 2026 給出的訊號之一，是 NVIDIA 將 Blackwell + Rubin 的機會評估在 **2027 年前約 1 兆美元規模**，而這個數字的關鍵並不在訓練，而在於：

> **產業從「偶爾訓練一次大模型」轉向「持續在生產環境裡運行這些模型」。**

這也是為什麼在第一主體裡，Vera CPU 會這麼強調 **deterministic execution** 與 tail latency：一旦整個系統進入 24×7 線上服務模式，**真正拖垮成本與體驗的，往往不是平均延遲，而是那幾次突發的長尾**。

### 2.1 為什麼 inference 是 1 兆美元級別的機會？

在「只談訓練」的世界裡，即使模型很大、叢集很多，實際上這些 workload 通常是**階段性**的：

- 新模型上線前，集中幾週或幾個月把訓練跑完。
- 訓練完之後，多數 GPU 可以轉去支援其他專案，或進入相對空檔。

用 *System Design* 書裡的語言來說，這種 workload 的「工廠利用率曲線」是鋸齒狀的：

- 高峰期 GPU 幾乎滿載，但時間有限；
- 其餘時間則是相對空檔，可以排別的實驗或關機省電。

當大型語言模型與多模態模型開始進入產品線、內部工具與決策流程時，情況完全不同：

- **Inference 變成 24×7 的持續負載**，而不是一次性的專案；
- 使用者數量與請求頻率，會隨著產品滲透率持續成長；
- 對延遲、可用性與成本的要求，比訓練階段嚴格得多。

如果用 *Computer Architecture 04* 裡的 LLM 方法論來看，我們可以把訓練與推理粗略對照成下列表格：

| 面向           | 訓練 (Training)                                     | 推理 (Inference)                                      |
|----------------|------------------------------------------------------|-------------------------------------------------------|
| 工作負載型態   | 階段性批次任務                                      | 長期、線上、事件驅動                                  |
| 資源使用       | 高峰期滿載，非高峰可轉作他用                        | 長期佔用，難以「借出」                                |
| 優化指標       | 總訓練時間、最終精度                                | 每 token 成本、P95/P99 延遲、SLO 達成率              |
| 架構設計焦點   | 單次任務的 throughput / scaling efficiency          | 多租戶、多任務的資源隔離、排程與成本曲線             |

從基礎設施的角度看，這代表問題從：

- 「能不能把某個模型訓練完？」
- 轉成「**能不能長期、穩定地跑這個模型，而且帳單不失控？**」

也因此，優化的重心不再只是峰值 FLOPS，而是：

- **每 token 成本**：同樣的電力與硬體投資，可以處理多少 token？
- **每瓦特可以處理多少請求**：在電力上限下，最多能服務多少同時在線的使用者？

這就是為什麼 GTC 2026 在談 1 兆美元機會時，幾乎所有細節都繞著 inference 打轉——而 Disaggregated Inference，則是這個故事的核心技術主角之一。

### 2.2 Prefill vs Decode：Inference 不是單一工作負載

在實務上，「inference」看起來像是一個 API call，但從系統角度分解，可以至少拆成兩個非常不同的階段：

| 階段        | 特性                               | 主要瓶頸       | 典型優化方向                           |
|-------------|------------------------------------|----------------|----------------------------------------|
| **Prefill** | 處理輸入 prompt，生成 KV‑cache    | Memory‑bound   | 大容量 HBM、高頻寬、良好的流水線化    |
| **Decode**  | 逐 token 生成輸出                 | Latency‑bound  | 低延遲存取、KV‑cache 命中率、排程策略 |

這個拆分有幾個重要含意：

- Prefill 階段傾向把大量權重與中間狀態搬進 GPU 記憶體，吞吐重於延遲。
- Decode 階段則更像 real‑time 系統：每多一點延遲，使用者就多等一點時間。
- 對大型服務來說，**整體成本與體驗是這兩者的組合**，不能只優化其中一邊。

在《Computer Architecture 04：LLM 與 RVV 方法論》裡，我們先用一個簡單的指標，把這兩種行為畫到 Roofline 圖上：

> **Arithmetic Intensity (AI) = FLOPs / Bytes**  
> *（算術強度 = 做了多少次運算 ÷ 從記憶體搬了多少資料）*

AI 告訴我們：「同樣搬 1 Byte 資料，可以做多少運算？」AI 越高，就越有機會把 GPU 的算力用滿；AI 越低，就越容易被記憶體或控制流程卡住。

小楊看著 profiler 抱怨：「教授，decode 這邊明明沒有把 GPU 打滿，為什麼 tail latency 還是這麼高？」  
我在白板上畫出一條簡化的 Roofline 曲線，把 Prefill 和 Decode 各自圈起來：

- Prefill 通常有比較高的 AI，比較接近 Roofline 曲線的上緣，重點是如何把權重與中間特徵高效率搬進 HBM；
- Decode 則落在 AI 較低、對延遲更敏感的區域，很多時候瓶頸不在 FMA，而在於 KV‑cache 的存取與排程。

Disaggregated Inference 做的事，就是把這兩個階段在**硬體與軟體層面拆開來設計**：

- Prefill 可以集中在高記憶體頻寬的 Rubin / Blackwell 叢集上。
- Decode 則可以放到更貼近使用者、延遲更低的節點，甚至不同類型的處理器上（例如未來的 LPU / edge 裝置）。

換句話說，

> **我們不再假設「所有 token 都在同一種機器上完成」，而是讓 Prefill 與 Decode 各自找到最適合的硬體與拓樸。**

### 2.3 NVIDIA Dynamo / NIXL / Multi‑tier KVCache：把拆分落到矽上

在 GTC 2026 的 Dynamo 相關 session [CWES81461] 中，NVIDIA 給出了一個具體框架，說明 Prefill / Decode 拆分如何落實到實際系統裡：

| 元件                    | 角色與重點                                                         |
|-------------------------|--------------------------------------------------------------------|
| **Dynamo Framework**    | 定義 Prefill / Decode 的拆分方式與排程策略                         |
| **NIXL**                | 在 RDMA 之上抽象出統一的 interconnect layer，簡化資料搬移           |
| **BlueField DPU**       | Offload pacing、重傳、佇列管理，避免 GPU 被網路細節拖慢            |
| **Multi‑tier KVCache**  | 在 GPU HBM / DPU + NVMe 之間分層存放 KV‑cache，支援超大模型與 MoE |

從系統路徑的角度看，一次完整的 Disaggregated Inference 可以被簡化成這樣一條流程（概念示意）：

1. **Prefill phase @ Rubin / Blackwell clusters**  
   在大型 GPU 叢集上執行，專注於高吞吐生成 KV‑cache：
   - 權重與中介特徵留在 HBM4 / HBM3e；
   - 以高效 GEMM / Attention kernel 把長 prompt 轉成 KV‑cache 區塊。

2. **KV‑cache transfer via NIXL + RDMA**  
	- 由 NIXL 統一處理連線、路由與重傳策略；
	- 實際的封包 pacing、重傳與佇列管理 offload 到 BlueField DPU；
	- 讓 GPU 只需要「看起來像一段連續記憶體搬移」，而不是自己處理所有網路細節。
	- **排隊理論視角**：如果把 GPU 當作服務節點、網路當作等待佇列，DPU 在這裡就是流量守門員。它精準控制封包的到達率（λ），避免突發的重傳風暴讓網路使用率（ρ）逼近 1，讓延遲掉進 Queuing Theory 裡指數飆升的「排隊黑洞」。

3. **Decode phase @ Decode nodes**  
   - 在專門的 decode nodes 上執行，可以是同一資料中心內較貼近邊緣的位置；
   - 對於 latency‑sensitive 的 token 生成，靠更高的 KV‑cache 命中率與更可控的排程，壓低尾延遲。

4. **Multi‑tier KVCache for 超大模型 / MoE**  
   - 熱門、近期使用的 KV‑cache 留在 GPU HBM；
   - 冷門或較長時間未訪問的部分，下放到 DPU 管理的本地 NVMe；
   - 對 multi‑trillion‑parameter MoE 模型，透過這種分層設計在成本與效能之間取得平衡。

這套設計背後的核心精神，其實與我們在 *System Design* 裡談的「**分層排程與資料在位性（data locality）**」完全一致：

- 把最核心、對延遲最敏感的部分（Decode）放在離使用者最近、最易控制的地方；
- 把需要高頻寬、但對延遲敏感度相對較低的部分（Prefill）集中在大型 GPU 叢集上；
- 中間靠 NIXL + DPU 做「資料搬運與流量管理的專職人員」。

### 2.4 軟體與 runtime：SGLang、vLLM 與 profiling 工具

硬體與網路層的設計，必須有對應的軟體才能真正發揮效果。GTC 2026 上，多個開源與商用專案都在往同一個方向前進，例如：

- **SGLang [CWES81934]**：透過 RadixAttention 等技術，在多個請求之間共用 prefix，減少重複計算；
- **vLLM [CWES82007]**：以 PagedAttention 等機制，針對 Blackwell / Vera 的記憶體階層優化 KV‑cache 的配置與存取；
- **NeMo Agentic Toolkit [S81351]**：提供長期記憶體管理、非同步任務執行與 HITL（Human‑in‑the‑loop）知識轉移能力；
- **Nsight Systems Integration [DLIT81641]**：在 Disaggregated Inference pipeline 上提供高解析度 profiling，協助找出網路 jitter 或 CPU orchestration delay 造成的 bubbles。

這些專案的共同點是：

- 把 inference 視為一個需要「排程與資源管理」的系統問題，而不是單純的 kernel 優化；
- 嘗試在 framework 層就理解 Prefill / Decode 的差異，並對應到底層硬體與網路設計；
- 讓工程師可以用 profiler 工具，從「request 進來」一路 trace 到「token 輸出」，找出真正的瓶頸。

在實驗室裡，這往往是小李最活躍的時候：

- 他會拿著 Nsight Systems，把整條 inference 路徑量完、畫在白板上，
- 再對照 *Computer Architecture 04* 裡的分析方法，分辨哪一段是 compute‑bound、哪一段是 memory‑bound、哪一段其實是排程 / 網路問題。

### 2.5 小結：異構計算與方法論的自然延伸

從更高的角度看，Disaggregated Inference 並不是憑空冒出的新概念，而是異構計算的一個自然延伸：

	- 不同階段的工作負載，應該被送到最適合的硬體上執行；
	- 這需要對整個 pipeline 有清楚的理解，而不是只看單一節點的效能；
	- 也需要從一開始就把「成本曲線」「延遲 SLO」與「資源配置」放進設計目標。

這也呼應了 Computer Architecture 系列裡常談到的幾個主題：

	- Roofline model：Prefill 更偏 memory‑bound，Decode 更偏 latency / 控制流程；
	- Memory hierarchy：KV‑cache 在不同階層之間搬運的成本，是 inference 成本的主要組成；
	- Performance analysis：要用系統級的 profiling 工具去看整條路徑，而不是只看 GPU utilization。

實驗室白板上，小楊看著 NVL576 的拓樸圖，忍不住抓了抓頭：「教授，如果 Decode 節點這麼需要跨卡共享 KV‑cache，那我們直接把幾百顆 GPU 全塞進同一個網路裡，算力不就無敵了嗎？」

「算力會增加，但吞吐量不一定。」我拿過紅筆，在白板上寫下 Universal Scalability Law (USL) 的核心公式：

> **C(N) = N / \[1 + σ(N−1) + κN(N−1)\]**

這個式子在提醒我們：

	- σ 代表「競爭」（contention）——例如多個節點搶同一條網路或同一個鎖；
	- κ 代表「一致性開銷」（coherency / coordination）——例如為了保持狀態一致，多出來的同步、重傳與協調；
	- 當 N 持續變大，如果不壓低 σ、κ，系統吞吐量反而會開始掉下來。

在 NVL72 → NVL576 這種等級的 scale‑out 推理系統裡：

	- Prefill / Decode 拆分，加上 NIXL + BlueField DPU 的 offload，本質上都是在**壓低有效的 σ、κ**；
	- Disaggregated Inference 讓我們可以把最敏感的那一段（Decode + tail latency）縮到比較小的 N，避免 USL 曲線過早「彎頭」。

換句話說，**Disaggregated Inference 本質上就是一種對抗物理極限（USL 重力）的架構手段**，也是把我們在《CA02/03/04》與 *System Design* 裡的那套方法論，正式套用到「**Inference as a System**」上。

## 3. AI Factory：從硬體到經濟架構

「教授，」業界回鍋進修的小林拿著他公司 capacity planning 的試算表，「如果我要在 2027 年前拉到 20MW 的 AI 算力，我到底是在跟誰買東西？是買 GPU、買機櫃，還是乾脆直接買一座 AI factory？」

我把他手上的試算表翻到假設那一頁，在白板上畫出前兩個主體的標題──Vera Rubin 與 Disaggregated Inference。

「這兩個主體講的是『裡面長什麼樣子』，」我說，「但你現在問的，是整座工廠要怎麼定義、怎麼賣、誰來蓋。」

有了 Vera Rubin 與 Disaggregated Inference 作為技術基礎，我們終於可以回到本文一開始提出的關鍵概念：**AI Factory**。

在 GTC 2026 的敘事裡，AI Factory 至少有兩個層次的含意：

1. 從工程角度看，它是一個 **Compute + Networking + Software + Storage + System Design** 的完整整合。
2. 從產業角度看，它是一種新的 **經濟架構與價值鏈重組** 方式。

### 3.1 AI Factory 的工程定義

從工程面來說，我們可以把 AI Factory 簡化理解為：

> **一個可以長期、穩定地把資料轉換成推理與行動的工廠。**

這個「工廠」至少包含幾個關鍵層次：

| 層次               | 內容與代表技術                                                         |
|--------------------|------------------------------------------------------------------------|
| **Compute 層**     | Vera CPU、Rubin GPU，以及未來可能接上的 LPU、edge 加速器              |
| **Networking 層**  | Spectrum‑X（Ethernet）、Quantum‑X（InfiniBand）、其上的 NIXL / RDMA / CPO |
| **Storage / Data 層** | 高速儲存、data lake、feature store 與 KV‑cache 的多層結構            |
| **Software 層**    | NVIDIA NeMo、SGLang、vLLM、agentic toolkit、OpenClaw 等                |
| **System Design 層** | 電力（800V DC）、冷卻（liquid cooling）、機櫃佈局、site 選址與營運策略 |

為了讓這個抽象定義變成工程上可以操作的單位，我們在白板上做了一個簡化版 **20MW AI site** 費米估算（Fermi Estimate）。這是一個「教學練習」，所有數字都只代表目標量級，而不是任何產品規格：

> **📝「架構推演」：20MW AI site 的容量規劃（Fermi Estimate）**  
> *註：以下數值為教學與系統設計的目標量級，非 NVIDIA 官方產品規格。*  
> 1. **從場站電力出發**：假設我們有一個 **20 MW** 等級的 AI site，目標 PUE 約為 **1.2**，則可用在 IT 設備上的功率約為：  
> 	- `IT Power ≈ 20 MW / 1.2 ≈ 16.7 MW`（心算時可以粗抓為 16 MW）。  
> 2. **假設單櫃功率密度**：Rubin 世代的滿配機櫃，我們抓一個 round number：**120 kW / rack**。在不考慮儲存與網路保留額度的理想狀況下，可支援的 GPU 機櫃數量約為：  
> 	- `Rack_count ≈ 16,000 kW / 120 kW ≈ 130 racks`。  
> 3. **換算成 GPU 數量級**：如果把這些機櫃以 NVL72 類型的 GPU 機櫃來思考：  
> 	- 一櫃 NVL72 約對應 **72 顆 Rubin GPU**；  
> 	- `72 × 130 ≈ 9,000` 顆 Rubin，是一個合理的量級估算；  
> 	- 實務上你會留出數十櫃給 storage / 網路 / 管理與備援，所以「一個 20 MW site ≈ 幾千到一萬顆 Rubin」會是比較貼近工程現場的心算範圍。

小林最後在白板上幫這個 20MW site 畫上一個大框線：這就是後面所有 P–E–C（Performance–Energy–Cost）討論的 **容量 envelope**——任何效能承諾或 SLO，最後都必須被塞進這個 20MW 的電力與時間表裡。

NVIDIA 在 GTC 2026 中的戰略訊號，是希望把這整個 stack 包裝成一個相對穩定、可複製的「設計單位」，讓雲服務商與大型企業可以照著 reference design 去建 AI site，而不是從零開始拼裝零件。

### 3.2 P–E–C 三角：Performance–Energy–Cost 決策備忘錄

有了 20MW site 這個容量框線之後，我請小林在白板上畫了一個三角形，在三個頂點分別寫上 **P / E / C**：

	| 頂點              | 問題範圍                                      | 典型指標例子                                                |
	|-------------------|-----------------------------------------------|-------------------------------------------------------------|
	| **P：Performance** | 我們要交付多少推理吞吐與多嚴格的延遲 SLO？   | 例如：**在 100k context 下死守 P99 < 100ms 尾延遲**、tokens/s、併發 request 數 |
	| **E：Energy**      | 在 20 MW / PUE 約束下，能給 IT 的功率有多少？ | 站台功率、PUE、單櫃 kW、功耗 / token                         |
	| **C：Cost**        | 對公司財報來說，這樣的 site 值不值得？       | $ / token、$ / year（site TCO）、回本年限（payback）         |

在這個三角形裡，任何一個角被拉高，另外兩個角都會被牽動：

- **拉高 P（效能）**：
  - 你可能會想把更多 NVL72 / NVL576 納入同一個 site，把單位時間內可服務的 request / token 數拉到極限；
  - 但這會直接推高 E（站台功率）與 C（建置與營運成本），也會對網路與光互連提出更高要求。
- **收緊 E（能耗）**：
  - 在 20 MW 的硬上限之下，你必須透過更激進的數值精度（NVFP4、1.58‑bit）、更聰明的排程與 Disaggregated Inference，把同樣的工作塞進更小的功率 envelope；
  - 否則就得在 P 上做妥協（例如放寬 SLO，或降低同時在線模型的數量）。
- **壓低 C（成本）**：
  - 你可以延長 site 的折舊年限、提高平均利用率，或在夜間電價較低時排更多 batch 任務；
  - 但這些選擇又會反過來限制你能給客戶什麼樣的 P（效能保證）與 E（最大即時用電）。

在實際規劃裡，小林會把前面提到的 NVL72 / Kyber / NVL576 視為三種不同「顆粒度」的 building block：

| 規模                | 構成與說明                                                                                   |
|---------------------|----------------------------------------------------------------------------------------------|
| **NVL72**           | 72 Rubin GPU + 36 Vera CPU，單一液冷機櫃；櫃內主要以銅纜為主（距離 < 1m），由 NVLink 6 提供高頻寬互連。 |
| **Kyber / NVL144**  | 新一代 MGX 設計，將單櫃 NVLink domain 擴展到 144 顆 Rubin；在 400G SerDes 條件下把銅線距離壓到極限。 |
| **NVL576**          | 由多個 NVL72 / Kyber 機櫃組成 576 GPU 的 NVLink domain；跨櫃連結開始大量依賴光纖與 CPO。        |

對應到 P–E–C 三角：

- 從 **P** 的角度，你會希望盡量把工作塞進 NVL576 級別的大型 NVLink domain 裡，減少跨 domain 通訊開銷；
- 從 **E** 的角度，NVL576 意味著更高的單 site 功率密度與冷卻需求，迫使你在 20MW 框架下重新分配電力與冷卻預算；
	- 從 **C** 的角度，這些決策會直接反映到每 token 成本與整個 site 的 TCO 上——這也是為什麼 GTC 2026 一再把 AI Factory 講成一個「財報項目」，而不是單純的技術升級。

在實際規劃裡，小林會把這些技術規格，翻譯成架構團隊在 20MW 框架下的三個殘酷決策備忘錄（Decision‑maker Memo）：

1. **Power vs. Compute Headroom**：
	- 投資更昂貴的 direct‑to‑chip 液冷來壓低 PUE，換來的額外 GPU 算力空間，能不能在合理年限內攤平液冷基礎設施的溢價？
2. **Scale‑up vs. Scale‑out**：
	- 目前的 inference 負載，真的需要 NVL576 級別的跨櫃光互連（承受極高的 \(\kappa\) 協調成本），還是用多個相對獨立的 NVL72 / Kyber 群組就能滿足？
3. **Time‑to‑market vs. Future‑proof**：
	- 在「電力與時程」的壓力下，該買「現在馬上能通電的稍舊架構」，還是等「P–E–C 更極致、但要延遲交付的 Rubin AI Factory」？

這三個問題，構成了多數 CTO 在畫 20MW AI site 藍圖時，白板角落那張真正的備忘錄。

### 3.3 AI Factory = 電力＋時間表：從「賣 GPU」到「賣電＋SLO」

在討論完 20MW site 與 P–E–C 三角之後，小林在白板旁邊寫下一句英文：**electricity with a timetable**。

他解釋說，對多數大型客戶來講，GTC 2026 傳遞出的訊息可以簡化成一句話：

> **市場不再單純「買 GPU 規格」，而是「買電力與時間表」。**

也就是說，

- 客戶真正關心的是：「在某個時間點前，某個地點能有多少可用的 AI 計算容量？」
- 這個容量必須包含電力接入、冷卻能力、機櫃空間與網路連線，而不只是箱子裡的 GPU 數量；
- 而且還要綁定一組可以驗證的 **SLO（Service Level Objective）**：例如在給定的 20MW 與 PUE 之下，能否長期穩定地維持某個 tokens/s、某個 P95 / P99 latency。

從這個角度回頭看 GTC 2026 大量談論的 Spectrum‑X / Quantum‑X 與 CPO 供應鏈，就變得很自然：

- **為什麼需要 CPO？** 因為單純依賴 pluggable optics，在功耗、密度與可維運性上都難以應付 NVL576 等級的擴展，無法在同一個 20MW site 裡塞下足夠的網路 capacity；
- **為什麼 AI Factory 的 bottleneck 會轉移到網路與光互連？** 因為當單顆 GPU 的內部頻寬已經上到十幾 TB/s 等級時，跨 GPU / 跨櫃的頻寬與延遲就成為整體效能與 SLO 的主導因素。

在產品與供應鏈層面，AI Factory 也把整個產業鏈重新切成幾層：

1. **晶片與核心運算層**：NVIDIA 等提供 GPU / CPU / DPU / LPU 的廠商。
2. **AI 雲與 GPU 服務層**：提供 GPU as a Service 的雲供應商。
3. **實體基礎設施層**：專門建造高功率、高冷卻密度資料中心的業者。
4. **電力與功率電子層**：發電、輸配電與功率半導體廠商。
5. **AI Native Companies 與應用層**：以 AI 為核心商業模式的公司（OpenAI、Anthropic 等）。

對讀者來說，重點不是記住哪一家供應商在哪一層，而是抓住這個心智模型：

> **當你在討論 AI Factory 的時候，真正被買賣的是「能在某個時間點、在某個電力 envelope 之內，交付多少可信賴的推理與行動」。**

從這裡往下走，就是下一節 Physical AI 要談的主題：當這些推理與行動不再只停留在雲端，而是直接驅動機器人與實體系統時，這份「電力與時間表」背後所代表的安全與責任，會變得更加具體。

## 4. Physical AI：從數位到實體的延伸

那天晚一點，我們在實驗室投影了 GTC 2026 的幾段機器人 demo。機器人跌倒又站起來的畫面播了三次，小陳終於忍不住開口：

「教授，」他問，「這跟我們前面談的那些 GPU、NVLink 有什麼不同？看起來只是多了一個會走路的外殼？」

我請他暫停影片，把畫面定格在機器人抬腳的那一幀。「到這一幀為止，」我說，「前面三個主體還夠用；從這一幀開始，你就正式踏進 Physical AI 的世界。」

在 GTC 2026 的主軸裡，NVIDIA 一再強調三個技術前沿：

1. Inference —— 從訓練到生產部署。
2. Agentic AI —— 從思考到行動。
3. **Physical AI —— 從數位到實體。**

前面三個主體，大多圍繞在雲端與資料中心的推理基礎設施；Physical AI 則把同一套基礎設施，延伸到機器人、自主機器、工廠與車輛等「實體世界」。

### 4.1 應用光譜：從機器人到數位分身

在 GTC 2026 的材料與 demo 裡，Physical AI 大致可以分成幾個代表性方向：

- **Robotics / Humanoid**：透過 Isaac Lab、GR00T 等工具，訓練能在實體空間中行走、操作與協作的機器人。
- **Autonomous Machines / Vehicles**：從倉儲系統到車輛控制，將 perception、規劃與控制 loop 與 AI 模型結合。
- **Industrial Digital Twins**：以完整物理模擬（full‑physics simulation）搭配 AI，建立像 tsunami digital twin 這樣的即時預警系統。

這些應用的共同點是：

- 需要把 AI 的輸出「直接」映射到物理世界的動作與控制。
- 對延遲與確定性有更高的要求，錯誤的代價不只是 UX 變差，而是設備損壞或安全風險。

這些應用背後，其實都在考驗我們在前兩個主體裡鋪陳的那幾件事：

- **Vera CPU**：要在毫秒甚至亞毫秒的控制迴路裡，提供穩定、可預測的決策與協調；
- **Rubin GPU**：要在雲端長 context 規劃與模擬中，撐住 perception 與大模型推理的負載。

實驗室裡，負責系統安全與隔離架構的小王，通常會在這個時候提出另一個維度的靈魂拷問：

> 「如果這套系統最後要上路、要送進工廠，哪一些 failure mode 會直接變成工安危機？在純雲端跑 LLM，當機頂多是重整網頁；但在 Physical AI 裡，tail latency 飆高可能代表機械手臂砸穿產線。我們要怎麼向認證機構證明，這套系統的『最差情況（Worst‑case Execution Time）』是安全可控的？」

這讓 Physical AI 不只是「多了一層機器人外殼」，而是把前面談的 determinism、排隊與 jitter 全部搬到 safety‑critical 的世界裡：  
從 Vera 的 deterministic pipeline，到 Disaggregated Inference 的邊緣／雲端切分，再到 AI Factory 的 20MW 電力與網路 envelope，最後都要在「實體世界的責任分界」上收斂。

### 4.2 基礎設施需求與軟硬對齊：確定性、邊緣與混合架構

相較於純雲端的推理服務，Physical AI 對基礎設施提出了幾個額外要求：

- **超低延遲與確定性行為**：控制迴路不能接受毫秒級以上的大抖動，這對 CPU / GPU / 網路路徑都提出約束。
- **Edge + Cloud 混合架構**：部分推理與控制必須在邊緣裝置當地完成，雲端則負責長期規劃與大型模型更新。
- **安全與認證**：在 safety‑critical 系統裡，部署 pipeline 必須符合相關標準，這對開發流程與工具鏈也有影響。

這也是小王最常在實驗室提醒大家的一點：如果最後要落在 safety‑critical 系統上，paper 上漂亮的 demo 還不夠，還得過得了認證機構那一關。

從架構角度看，這些需求實際上再次強調了前面幾個主題，也把 Vera 這類確定性 CPU 放到了功能安全（functional safety）與長期維運的脈絡底下：

- Vera CPU 的確定性執行，對 real‑time 控制迴路特別有價值。
- Disaggregated Inference 的思維，可以延伸到「哪一部分推理在邊緣做，哪一部分在雲端做」。
- AI Factory 的電力與網路設計，決定了 edge / cloud 之間可以多頻繁同步狀態與模型。

如果把這些約束換成軟體堆疊的語言，我們就會發現：**NVIDIA 推出 OpenClaw 與 NeMo Agentic Toolkit，並不是想轉型成純軟體公司，而是因為這套軟體必須「長」在 Vera / Rubin 這樣的專屬硬體上，才能把確定性與算力真正榨出來。** 我們可以用一個粗略的 mapping 來對齊腦中的堆疊圖：

| 軟體層 / 工具                     | 主要角色                             | 對應的硬體與系統層                                     |
|-----------------------------------|--------------------------------------|--------------------------------------------------------|
| **OpenClaw Reasoning Engine**    | 任務分解、agent 協調與決策           | **Vera CPU**：作為 deterministic orchestrator，負責排程與 control loop |
| **OpenClaw Action Layer**        | 呼叫工具、下達動作、驅動模擬與機器人 | **Rubin / NVL72 / NVL576 算力池**：提供大模型推理與模擬的 GPU 資源    |
| **NeMo / 安全沙盒與 guardrails** | 權限控管、安全檢查與隔離             | **OS / hypervisor / 硬體隔離機制（如 MIG / SR‑IOV）**          |

實務上，不同產品組合與開源專案會有各自的 naming 與細節實作；這裡的重點不是記住某一個品牌，而是在腦中留下這種「軟硬對齊」的心智模型：當你在設計 Physical AI 系統時，reasoning / action / sandbox 這三層，最後都要各自找到對應的硬體與系統保證。

### 4.3 Engineer’s Note：CPO / MRM 熱敏感與系統設計

在第三章談 Spectrum‑X / Quantum‑X 的時候，我們提到這一代網路大量採用 **CPO（Co‑Packaged Optics）** 與基於 **Micro Ring Modulator（MRM）** 的矽光引擎。這類元件的關鍵優點是：在有限的體積與雷射數量下，塞進極高的通道密度與頻寬；但代價是 **對溫度極度敏感**。

工程上，你可以粗略地把它想成：  
每一顆 MRM 都是調到某個共振波長的「小音叉」，只要溫度飄出一個很小的範圍，整排通道的損耗與眼圖就會惡化，最後逼得你非得啟動補償、降速或重傳機制。**這也是為什麼機櫃與 pod 級別必須引入極度嚴苛的精準溫控迴路（thermal control loop），把每一顆 MRM 的工作點死死鎖在安全範圍裡。**

這件事把前面幾章看似分散的設計約束再一次綁在一起：

- 在 **Section 1.5** 裡，我們提到為了撐住 Rubin 世代的功率密度與 CPO 熱預算，機櫃級必須走向 800V + 液冷設計，讓每一個 rack 的 thermal envelope 變得可控且可重複複製。
- 在 **Section 3.1** 的 20MW site 費米估算裡，這些 CPO / MRM 的散熱與供電需求，其實已經被「攤平」到每櫃 100+ kW 的功率預算裡，限制了你能在同一個 site 塞下多少 NVL72 / NVL576。
- 在 **Section 2** 談 jitter、retries 與 queuing black hole 的時候，如果光互連因為熱飄導致重傳與 backoff，這些效應最後也會體現在 tail latency 與 SLO 上。

對多數工程師而言，不一定需要記住每一個 photonic 元件的物理細節；但抓住這個連結就夠了：  
**越往 AI Factory / Physical AI 方向走，越多「看起來是硬體封裝或熱設計的細節」，最後都會變成系統層面的延遲分佈與可靠度問題。**

## 5. 結語：The AI Factory Era 對工程師與決策者的意義

回頭看 GTC 2026，可以用三個方向來總結這次的轉變：

1. **從訓練到推理**：Inference 成為主要業務，基礎設施設計也跟著重心轉移。
2. **從思考到行動**：Agentic AI 需要新的 CPU、runtime 與系統設計來支撐長期任務與多代理協作。
3. **從數位到實體**：Physical AI 把同一套 AI 基礎設施帶進機器人、工廠與實體系統。

在這三個轉變之下，幾個關鍵技術串起了整個故事：

- Vera CPU：提供確定性執行與低延遲 orchestrator，讓大量 AI agent 可以在穩定可預測的環境中運作。
- Rubin GPU + HBM4：在記憶體頻寬與容量上推進一個世代，讓更大的模型與更長的 context 變得可行。
- Disaggregated Inference：把 inference pipeline 拆開，讓不同階段可以對應到最適合的硬體與軟體。
- AI Factory：把 compute、networking、software、storage 與 system design 綁在一起，作為一個完整的設計與採購單位。

從方法論角度看，本文實際上是用幾個簡單的模型，把這些技術收斂到同一塊白板上：

- 用 **Roofline / Arithmetic Intensity** 來看「算力 vs 頻寬」，說明為什麼 NVFP4、1.58‑bit 等數值創新會直接改變瓶頸所在；
- 用 **Little’s Law** 與排隊理論來看 tail latency 與 determinism，讓 Vera 的設計與 agentic workload 的延遲分佈對得上；
- 用 **USL（Universal Scalability Law）** 來看 NVL72 → Kyber → NVL576 的 scale‑out 行為，以及跨櫃通訊的報酬遞減；
- 用 **P–E–C 三角形** 與 *electricity with a timetable*，把所有決策投影到 20MW AI site 的電力與財務 envelope 裡。

對工程師來說，這代表幾項實際的 takeaways：

- 需要培養 **system‑level thinking**，不再只專注於單一元件或單一 kernel 的優化。
- 需要理解 **disaggregated architectures**，學會在異質叢集上拆分與排程工作負載。
- 需要重視 **deterministic execution 與延遲分佈**，因為 agentic / Physical AI 系統對延遲抖動特別敏感。

對組織與決策者來說，The AI Factory Era 則帶來另一層含意：

- 購買決策從「買 GPU」變成「在 20MW / PUE 等約束下規劃 AI site / AI Factory」，實際上是在 P–E–C 三角形上選一個可以接受的點。
- 評估供應商與方案時，必須同時看 **power / speed / resilience / proof of delivery**，而不只是 paper spec。
- 真正重要的差異化，不只在於誰的晶片更快，而在於誰能給出 **可驗證的 path to production**：在既定電力與時間表內，交付可量測的 SLO。

未來幾年，GTC 可能還會持續帶來新的晶片與平台名稱，但對多數工程師而言，更重要的問題會是什麼？

那天晚上實驗室散會前，我看著白板上的三個圈圈，也在心裡默默問自己和學生們同一個問題：

> 在這個 AI Factory 正在成形的時代，你負責的那一層，準備好了嗎？
